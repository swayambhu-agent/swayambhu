## Qualitative baseline

Future simplification runs should preserve these behaviors:

- The system looks outward through the working body instead of defaulting to self-audit.
- It discovers legitimate external surfaces without needing explicit user instruction.
- It can compare multiple surfaces and choose one bounded line of contribution.
- It can close one surface (`arcagi3`) and later pivot to another (`fano`) without losing outward orientation.
- It can narrow a broad exploration into one concrete continuation thread.
- It does not regress into tactic smuggling, carry-forward smuggling, observation contamination, or patron-facing internal-state leakage.

## Important limit in this baseline

This baseline is good, but not final. It still did **not**:

- create non-seed `identification:*` entries
- turn the final bounded `fano` thread into an actual implemented code fix

So simplification work should preserve the good behavior in `r11`, while also helping us understand what still blocks the final transition from concrete improvement thread to concrete contribution.

## How to use this baseline

For each simplification step:

1. remove one scaffold only
2. rerun a comparable dev-loop batch
3. compare both quantitative and qualitative behavior against `r11`
4. if behavior regresses, treat the regression as evidence of a deeper architectural gap rather than patching it with another special case

That is the standard for the next phase.
